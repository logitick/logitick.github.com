Struggling with OOP

Paul Daniel Iway · 2019.5.26

In a previous job interview, I’ve been asked a basic question about OOP: “What are the four pillars of OOP?“. It made me feel really bad about not being able to give an answer because I use these concepts everyday.

The interview had also reminded me of how frustrating it felt to not understand the concept during my time in college. Those books where they tell you that a cat is an object didn’t help at all. Using cats and dogs as objects does not make it easier for a someone struggling to get a gist of what an object is.

I figure it’s time for a refresher.

Since we’re builing software, I’ll be using real world examples.

What is an object?

Let’s pretend we’re working for a e-commerce company and we’ve been asked to work on processing the payments.

In this system we could have a CreditCard object. We could also have an Invoice object. Some other objects we could add are SalesOrder or Product.

An object is a representation of real-world objects inside our system. Although, they’re not limited to items in the real-world (e.g. data structures).

Class vs Object

Hopefully you now get the gist of what an object is. But what is a class and how does it differ from an object?

To put it simply: a class describes the object. Still confused? In the next section we’ll describe a Credit Card:

Properties

What makes a Credit Card? You can’t have a credit card without a credit card number, expiry date, a cvv, and the card owner’s name. These make our class’ properties.

class CreditCard {
    public String cardNumber;
    public String expiryDate;
    public String cvv;
    public String ownerName;
}

Disclaimer: For the sake of simplicity the properties here are public. This is a code smell and you should avoid exposing your class properties.

Methods

What can we do with a credit card? We can charge the card or save it for later. These are the methods. A method is something we can do to an object. Usually methods names are actions.

class CreditCard {
    public String cardNumber;
    public String expiryDate;
    public String cvv;
    public String ownerName;

    public void charge(Integer amount) {
        ...
    }

    public void save() {
        ...
    }
}

Why do we need to do this? It’s too complicated for itty-bitty pieces of data. Why not just have variables for the creditCardNumber, cvv, expiryDate, and ownerName. -Imaginary guy who has just learned to code.

The thing is we need to pass this data around the system and passing those 4 variables from one function to another could lead to mistakes. Oh, and what if you were dealing with more than 4 variables, say an Invoice.

Also, when (not if) the spec changes you’d have to look all over for those variables.

A class wraps all those variables for you which makes life a little bit easier. In the OOP world this is called encapsulation which I would like to talk about some more in a separate post about the four pillars of OOP.
